# Travel Atlas 設計文件

## 背景與目標

過去旅遊紀錄分散在 Apple 備忘錄（真正的日記/心得，無 API 可讀）與 Notion（行前規劃筆記：待辦、交通、景點研究）。使用者看到他人用 Obsidian + Claude Code 做出的「Travel Atlas」視覺化頁面（3D 地球儀、旅行統計、地區分佈、年度足跡、可篩選旅程清單），想做出類似效果，但資料改用 Firebase 即時存取，讓手機上隨時可以新增日記/照片，而不是編輯 markdown 檔案後重新生成。

## 範圍

- 建立單檔 HTML 手機優先 PWA，架構與現有 `trip-template.html`／`japan-trip.html` 一致
- 首頁做「Travel Atlas」視覺化總覽（統計、地圖、圖表、旅程清單）
- 每趟旅程有時間軸式日記（日期、地點、文字、照片、分類標籤）
- 一次性把 Notion 已成行旅程的背景資訊 + 使用者提供的 Apple 備忘錄內容，整理成初始資料匯入
- 匯入完成後，Notion／備忘錄不再是紀錄旅遊的地方，全部改用新 app

不在範圍內：行前規劃功能（待辦、交通比價等，仍留在 Notion 作為既有工具）、自動化測試、3D WebGL 地球儀（先做 2D 地圖，效能穩定後再評估）。

## 架構

- **前端**：純 HTML/CSS/JS 單檔 PWA，Tab 架構（Atlas 首頁 / 旅程列表 / 統計），FAB 新增日記，可加到手機主畫面
- **後端/儲存**：Firebase（Firestore 存文字資料、Storage 存照片），與現有 app 共用同一個 Firebase 專案
- **地圖**：2D 世界地圖 + 發光城市光點（走過的城市），可點擊 marker 跳到對應旅程；不用即時定位或路線規劃
- **離線處理**：新增日記先寫入 localStorage，偵測到網路後同步至 Firestore，避免旅行中收訊差時遺失資料

## 資料模型

### Trip（Firestore `trips` collection）

| 欄位 | 說明 |
|------|------|
| `title` | 旅程標題，例：「2023/11 沖繩」 |
| `destination` | 目的地文字 |
| `region` | 地區分類（東亞/東南亞/歐洲/美洲/南亞/大洋洲…），用於圓餅圖與篩選 |
| `startDate` / `endDate` | 旅程起訖日期 |
| `tags[]` | 標籤 |
| `coverPhotoUrl` | 封面照（Storage URL） |
| `status` | 固定為 `visited`（已成行） |
| `countryCode[]` | 國家代碼陣列，用於地圖標記與國家計數 |

### Entry（`trips/{tripId}/entries` subcollection）

| 欄位 | 說明 |
|------|------|
| `date` | 日記日期 |
| `location` | 地名 + 經緯度（可選） |
| `text` | 日記內文 |
| `photos[]` | 照片 Storage URL 陣列 |
| `category` | 分類（美食/景點/住宿/交通…） |

統計數字（旅行趟數、國家數、起始年份、地區分佈、年度足跡長條圖）皆從 Trip 文件即時計算，不另外存欄位。

## 畫面結構

### 首頁「Travel Atlas」
- 頂部統計列：旅行趟數 / 國家數 / 累積年數
- 2D 世界地圖：發光光點標記走過的城市，可拖曳/點擊 marker 跳到對應旅程
- 地區分佈圓餅圖 + 年度足跡長條圖
- 旅程清單：可依地區篩選，每筆顯示國旗、標題、日期、地區；點擊進入旅程詳情

### 旅程詳情頁
- 封面照 + 標題 / 日期 / 地區 / 標籤
- 時間軸式日記列表（依日期排序），每則顯示地點、文字摘要、縮圖；點開看完整內容與多張照片

### 新增日記（FAB）
- 選擇既有旅程或建立新旅程
- 填寫日期 / 地點 / 文字 / 拍照或選照片上傳 / 選分類標籤

## 資料遷移（一次性）

1. **Notion**：抓取「瑜的旅遊計畫」下已成行的旅程頁面（沖繩、東京櫻花、墨爾本、多羅米蒂、東京熱海、京都楓葉、長野雪地健行、芬蘭極光等），萃取 `destination`、大略日期、標籤等 Trip 層級背景資訊，作為初始 Trip 文件的基礎。未成行/純研究性質頁面（阿里山、機票教學、滑雪等）不納入。
2. **Apple 備忘錄**：真正的日記/心得/照片來源，無 MCP 可讀，需使用者手動提供文字內容（複製貼上）與照片（直接傳送或放入可讀取的資料夾）。
3. 兩邊資料合併整理成統一格式，寫入 Firestore 作為初始資料。完成後 Notion／備忘錄的旅遊紀錄職責收掉，後續一律用新 app 記錄。

## 錯誤處理

僅處理會實際發生的情境：照片上傳失敗、Firestore 寫入失敗時提示使用者重試。不做其他防禦性設計。

## 測試

匯入完成後，用 Playwright 手動走過新增日記、瀏覽地圖、依地區篩選旅程等主要流程，確認可用。不寫自動化測試（除非之後有需要再加）。
